Чеклист проверки данных для документа **Тяжесть** (`heaviness`, шаблон `heaviness-protocol.docx`).

Применяй ВМЕСТЕ с универсальными проверками типографики и полей из `check.md`.
Корень JSON: `protocol / customer / measurementDate / workplaces[] / performer / representative`.
Каждая карточка рабочего места содержит ~17 показателей вида `pN_…` со структурой `{ value, class }`.

## Обязательные поля (пустая строка = ошибка)
- `protocol.number / year / day / month / dateYear`
- `customer.name`, `customer.address`
- `measurementDate.day / month / year`
- `performer.fullName`, `performer.position`
- `representative.fullName`, `representative.position`
- В каждом `workplaces[]`: `code`, `position`, `measurementPlace`, `workDescription`, `finalAssessment`
- Каждый показатель `pN_…` обязателен и должен содержать поле `class` (его `value` может быть пустым)

## Формат числовых полей
- `class` у каждого показателя и в `finalAssessment` — строго из набора **`"1"`, `"2"`, `"3.1"`, `"3.2"`** (разделитель — **точка**, не запятая; `"3,1"` и `"5"` — ошибка). Это ýже общего набора классов: классов `3.3 / 3.4 / 4` в тяжести НЕТ.
- `value` — свободная строка (число, диапазон или текст), может быть пустой.
- `rowNumber` — целое положительное; коды рабочих мест в формате `01 NNN NNN` (см. кодировку).

## Что должно быть подчёркнуто
Вставляемые значения в готовом .docx:
- `protocol.number`;
- наименование и адрес заказчика;
- даты;
- ФИО и должности исполнителя и представителя.

## Типографика этого документа
- Месяцы — родительный падеж, строчными (`апреля`).
- `measurementPlace` — название раздела с тире «–» и пробелами, без точки в конце.
- `position` — должность как в кодировке (одинаковое написание во всех протоколах).
- Итоговый класс `finalAssessment` должен соответствовать максимальному классу среди показателей.
- `customer.name` — в «ёлочках».
