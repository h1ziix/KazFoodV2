Чеклист проверки данных для документа **Кодировка** (`coding`, шаблон `coding-protocol.docx`).

Применяй ВМЕСТЕ с универсальными проверками типографики и полей из `check.md`.
Корень JSON: `approval / sections[]`. Это единственный документ без блока `protocol`/`customer`.
Кодировка — источник правды по кодам рабочих мест для всех остальных протоколов.

## Обязательные поля (пустая строка = ошибка)
- `approval.position`, `approval.organization`, `approval.fullName`
- `approval.date.day / month / year`
- В каждом `sections[]`: `title` и минимум одна строка
- В каждой строке: `name` (наименование должности)

## Формат числовых полей и кодов
- `count` — **только `0` или `1`** (целое). Любое другое значение — ошибка.
  - `1` — аттестуемое рабочее место (получает код);
  - `0` — справочная должность, исключается из протоколов, код пустой.
- `code` — **производное значение, назначается автоматически**, вручную не правят:
  - при `count = 1` строго формата `01 NNN NNN` (`/^01 \d{3} \d{3}$/`, одиночные пробелы) — например `"01 001 001"`;
  - при `count = 0` код обязан быть пустой строкой `""`.
- `sections[].number` — целое положительное; третий блок кода в каждом разделе начинается с `001`.
- Повторяющиеся должности — **отдельными строками**, а не `count > 1` (например, два «Менеджера по продажам» → строки 006 и 007).

## Что должно быть подчёркнуто
Вставляемые значения в блоке утверждения готового .docx:
- `approval.fullName`, `approval.position`, `approval.organization`;
- дата утверждения.

## Типографика этого документа
- `title` раздела — без ведущего номера и без точки в конце; тире «–» с пробелами.
- `approval.organization` — в «ёлочках».
- `approval.fullName` — `Фамилия И.О.` (инициалы с точками).
- Месяц в дате — родительный падеж, строчными.
- Наименования должностей должны совпадать с теми же должностями в остальных протоколах (синхронизация по коду).
