Чеклист проверки данных для документа **Травмобезопасность** (`safety`, шаблон `safety-protocol.docx`).

Применяй ВМЕСТЕ с универсальными проверками типографики и полей из `check.md`.
Корень JSON: `protocol / customer / measurementPlace / measurementDate / sections[] / performer / representative`.
Внимание: у этого документа `protocol` содержит **только** `number` (нет year/day/month).

## Обязательные поля (пустая строка = ошибка)
- `protocol.number`
- `customer.name`, `customer.address`
- `measurementPlace`
- `measurementDate.day / month / year`
- `performer.fullName`, `performer.position`
- `representative.fullName`, `representative.position`
- В каждом `sections[]`: `title` и минимум одна строка
- В каждой строке: `code`, `position`, `documentation`, `result`, `nonComplianceReasons`
- `equipment` и `finalNote` — **необязательные**; пустые допустимы

## Формат числовых полей
- `count` — целое число ≥ 0. Здесь, в отличие от кодировки, `count` **может быть больше 1**: повторяющиеся должности свёрнуты в одну строку с количеством (например, «Сторож», 9).
- `code` — построчный код раздела в формате `01 NNN NNN` с одиночными пробелами; нумерация строк раздела без пропусков (001, 002, 003…).
- `sections[].number` — целое положительное.

## Значения служебных полей по умолчанию (проверь, что не искажены)
- `documentation`: `"в наличии"`
- `result`: `"соответствует"`
- `nonComplianceReasons`: `"отсутствуют"`
- `finalNote`: `"соответствует стандартам"`

## Что должно быть подчёркнуто
Вставляемые значения в готовом .docx:
- `protocol.number`;
- наименование и адрес заказчика;
- даты;
- ФИО и должности исполнителя и представителя.

## Типографика этого документа
- `title` раздела — БЕЗ ведущего номера («1. ») и без точки в конце: генератор сам подставляет `N. ` при рендере. Если в JSON уже стоит «1. Администрация…» — это задвоение.
- Тире в названиях разделов — «–» с пробелами.
- Месяц — родительный падеж, строчными.
- `position` — как в кодировке. `customer.name` — в «ёлочках».
