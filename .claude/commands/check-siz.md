Чеклист проверки данных для документа **СИЗ** (`siz`, шаблон `siz-protocol.docx`).

Применяй ВМЕСТЕ с универсальными проверками типографики и полей из `check.md`.
Корень JSON: `protocol / customer / measurementPlace / measurementDate / sections[] / performer / representative`.
Внимание: `protocol` содержит **только** `number`.

## Обязательные поля (пустая строка = ошибка)
- `protocol.number`
- `customer.name`, `customer.address`
- `measurementPlace`
- `measurementDate.day / month / year`
- `performer.fullName`, `performer.position`
- `representative.fullName`, `representative.position`
- В каждом `sections[]`: `title` и минимум одна строка
- В каждой строке: `code`, `position`, `issuedFact`, `certificate`, `assessment`, `note`
- `normItems` («Нормированный перечень») — **необязательное**; пустое допустимо

## Формат числовых полей
- `count` — целое число ≥ 0 (в примере у всех строк `1`).
- `code` — код в формате `01 NNN NNN` с одиночными пробелами, нумерация строк раздела без пропусков.
- `sections[].number` — целое положительное.

## Значения служебных полей по умолчанию (проверь, что не искажены)
- `issuedFact`: `"да"`
- `certificate`: `"в наличии"`
- `assessment`: `"обеспечен"`
- `note`: `"-"` (прочерк, а не пусто)

## Что должно быть подчёркнуто
Вставляемые значения в готовом .docx:
- `protocol.number`;
- наименование и адрес заказчика;
- даты;
- ФИО и должности исполнителя и представителя.

## Типографика этого документа
- `title` раздела — БЕЗ ведущего номера («1. »): генератор подставляет `N. ` сам. В примере заголовок может нести количество мест («Администрация – 3 рабочих мест») — следи за согласованием числа и тире «–».
- Месяц — родительный падеж, строчными.
- `position` — как в кодировке. `customer.name` — в «ёлочках».
